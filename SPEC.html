1. Introduction

1.1 What is Rutile?

Rutile is a graphics API that occupies the layer between user-facing frontends and low-level platform graphics APIs. It defines a common backend interface that multiple conforming implementations can satisfy, permitting a frontend to be written once and run on any conforming backend.

Rutile is a user-space API. It does not interact with graphics hardware directly and does not define driver-level behavior. Implementations sit on top of driver-space APIs such as Vulkan, Metal, or Direct3D 12, or on top of software renderers.

This specification describes the externally visible behavior that a conforming backend implementation must exhibit. Internal caching, deferred work, platform integration, memory management strategy, and command dispatch mechanisms are unconstrained except where they affect externally visible results.

1.2 Design goals

1.2.1 Object model

Rutile adopts an object model in the tradition of OpenGL. Resources are opaque objects referenced through handles, operations on those objects are expressed as function calls, and the lifetime of each object is explicit. This model maps cleanly onto most language runtimes and is straightforward to wrap.

1.2.2 No global bind state

Rutile departs from OpenGL by eliminating global bind state. In OpenGL, resources must be bound to global slots before they can be used; this couples unrelated operations, complicates reasoning about resource state, and introduces hidden dependencies between call sites. In Rutile, resources are passed directly to the functions that operate on them. No global binding table exists.

1.2.3 Abstraction-friendly interface

Rutile does not attempt to be a convenient end-user API. It assumes that a frontend — a framework, a wrapper library, or a set of language-specific bindings — will sit between Rutile and application code. Design choices are made to make that frontend layer as straightforward to write as possible, not to make raw Rutile calls pleasant to write directly.

Concretely: ownership and lifetime rules are specified precisely enough that a RAII wrapper can be written without ambiguity; error state is exposed in a form that is easy to check and forward; implicit behavior that would be difficult for a frontend to intercept or override is avoided.

1.2.4 No global state beyond initialization

With the exception of the backend instance and the thread-local error state, Rutile defines no global state. All resource state is owned by individual objects and is reachable only through valid handles. This constraint prevents hidden interference between independent call sites within the same process.

1.3 Scope of the backend interface

1.3.1 Target platforms

A conforming Rutile backend MUST be implemented on top of one or more of the following: Vulkan, Metal, Direct3D 12, or a software renderer. OpenGL is not a supported implementation target. Its global bind model is structurally incompatible with Rutile's resource design; supporting it would require either emulating global bind state in the backend or violating Rutile's object model guarantees.

1.3.2 Function loading

Rutile does not define how its functions are loaded. Whether a backend is linked statically, loaded as a shared library, or resolved through a custom loader is outside the scope of this specification. Each function has a canonical exported symbol name; how that name is resolved at runtime is the responsibility of the loader.

1.3.3 Frontend responsibility

Rutile does not define a frontend. Language bindings, RAII wrappers, render-graph systems, and higher-level scene abstractions are frontend concerns. This specification defines only what a backend must do and places no requirements on anything that calls into a backend.

1.4 Scope of this specification

This specification defines the data types and object state, the resource model, lifecycle rules, initialization requirements, validation-layer semantics, and error behavior of the backend function interface.

This specification does not define rendering algorithms, performance requirements, internal implementation strategy, threading guarantees beyond those explicitly stated, function loading mechanisms, or frontend design.

2. Conventions and definitions

2.1 Normative language

The key words MUST, MUST NOT, and MAY are used as defined below. Statements using these words define required, prohibited, or optional behavior of a conforming implementation.

MUST

indicates required behavior. A conforming implementation must satisfy this constraint unconditionally.

MUST NOT

indicates prohibited behavior. A conforming implementation must never exhibit this behavior.

MAY

indicates optional behavior. A conforming implementation is permitted but not required to exhibit this behavior.

2.2 Terms and notation

throw an error

to set the thread-local error value to a non-success error code

throw a message

to write a string into the thread-local error message buffer

2.3 Function specification structure

Each function specification uses the following fields. All fields are present for every function; a field with no applicable content is marked None.

Requirements

observable backend behavior required by this specification. Requirements are not validation obligations unless the requirement explicitly says the backend must diagnose the condition.

Pre-conditions

conditions that must hold before the call is made. If they do not hold, behavior is undefined. The backend is not required to detect, report, or recover from violated pre-conditions.

Post-conditions

guarantees that hold if and only if the operation succeeds.

Errors

the complete set of error codes this function may throw.

Default

the value returned when the function throws. Applies only to functions with a non-void return type.

2.4 Behavior categories

2.4.1 Undefined behavior

Undefined behavior is behavior for which this specification imposes no requirements on the implementation. An implementation is not required to detect, diagnose, or handle undefined behavior in any particular way.

2.4.2 Race condition

Concurrent access to shared state that does not satisfy the synchronization rules stated by this specification constitutes a race condition and is undefined behavior.

2.5 Operations

An operation is any Rutile function call that reads, writes, creates, destroys, or otherwise interacts with backend-managed state. Operations execute in the order issued unless this specification explicitly states otherwise. An operation begins when the call starts execution and ends when it returns, unless a specific operation states otherwise.

2.6 Backend instances

A backend instance is a single isolated execution context of the Rutile implementation. All valid handles are scoped to the backend instance that created them. A backend instance begins when Rutile is initialized and ends when it is shut down. The active backend instance is global backend-managed state. This specification currently defines no direct interface for interacting with backend instances; the frontend manages the instance conceptually.

2.7 Concurrency

Unless explicitly specified as thread-safe, any Rutile function call that operates on a given handle must not execute concurrently with any other Rutile function call that operates on the same handle. Concurrent operations on disjoint handles are permitted unless they share underlying backend-managed state (for example, two operations that both write to the same framebuffer).

Concurrent operations on the same handle are undefined behavior unless every concurrently called operation is explicitly specified as thread-safe for that handle.

3. Errors

3.1 Error model

Rutile uses a thread-local error state of type rt_error and a thread-local error message buffer. Operations report failure by setting the thread-local error state rather than returning an rt_error directly, unless a specific function in this specification explicitly returns rt_error.

The error state is sticky. Successful operations are not required to clear a previous error. Callers that need a known baseline must call rtClearError() before issuing the operation they want to observe.

On failure, an operation writes an appropriate non-success error code. An operation MAY also write a diagnostic message. Message contents are implementation-defined unless a specific function states otherwise.

Validation layers MAY validate pre-conditions and write RT_IMPROPER_USAGE when violations are detected. The backend is not required to detect invalid handles, null pointers, invalid enum values, invalid object state, or invalid call ordering unless a function explicitly defines such detection as backend behavior.

3.2 Error accessors

The functions rtError and rtErrorMessage are the primary means of reading error state. They are free functions and are specified in full in section 6.

3.3 Error message buffer

The backend MAY write to the thread-local error message buffer when it reports an error. Before any operation writes a message, the buffer contains an empty string.

When a message is written, it is null-terminated and stored in implementation-owned thread-local storage. The Vulkan backend stores at most 1023 message bytes plus the null terminator.

Where this specification requires a message but does not specify its exact contents, the content is implementation-defined.

4.2 Handles

All handle types are type-safe opaque resource identifiers with a _h suffix, such as swapchain_h or framebuffer_h. Every resource type has a corresponding handle type. From the perspective of Rutile users, all handles have the representation of an unsigned 64-bit integer (u64), but their internal representation is implementation-defined.

4.2.1 Reserved values

The value 0 is reserved and represents null. Any non-zero value is invalid unless it was returned by a create function — either directly or implicitly through a successful create function that creates backend-owned child resources — and has not been destroyed.

4.2.2 Validity rules

A handle is valid if and only if all of the following are true:

  • it was returned by a successful Rutile function call that created a resource, either directly or implicitly
  • the resource has not been destroyed
  • the handle is used within the backend instance that created it

A handle is invalid if any of the following are true:

  • it was not returned by a successful resource-creating Rutile function call, either directly or implicitly
  • the resource it referred to has since been destroyed
  • it is used with a different backend instance

Null is not valid. Using null as though it referred to a valid resource is undefined behavior unless a function explicitly defines behavior for a null argument. Use of any invalid handle is undefined behavior.

4.2.3 Identity and reuse

A handle value has no persistent identity beyond the lifetime of the resource it represents. Each valid resource MUST be associated with exactly one unique handle value within the backend instance. While a resource exists, its handle value MUST be unique within that instance. The backend MAY reuse a handle value after the previous resource associated with it has been destroyed. Reuse MUST NOT affect any currently existing resource.

Handle comparison is defined behavior only when both operands are null or are valid handles of the same type. Comparing an invalid handle is undefined behavior. Two handles sharing the same numeric value do not imply identity: if handle A is destroyed and a subsequent create call returns a new handle that happens to share A's numeric value, the two handles name distinct resources. Equality of numeric values across distinct resource lifetimes has no defined meaning.

4.2.4 Opaqueness

Any interpretation of a handle value other than null-check or same-type equality comparison is undefined behavior. No bit pattern, numeric range, ordering, or encoding within the handle value has defined meaning to Rutile users.

5. Initialization and shutdown

5.1 Backend state

The backend has exactly two global states: Uninitialized and Initialized. In the Uninitialized state, calling any Rutile function other than rtInit(), rtGetName(), and rtError() is undefined behavior.

5.2 Initialization

void rtInit( u32 ext_count, const char* const* extensions )

Parameters:

  • ext_count — the number of extension name strings in extensions. May be 0.
  • extensions — a pointer to ext_count null-terminated UTF-8 extension name strings identifying the extensions to enable. May be null when ext_count is 0.

This function MUST:

  • transition the backend to the Initialized state so that it is ready to accept all Rutile function calls
  • set the error state to RT_SUCCESS on success
  • leave the backend in the Uninitialized state if initialization fails

Errors: RT_ALREADY_INITIALIZED, RT_OUT_OF_HOST_MEMORY, RT_OUT_OF_DEVICE_MEMORY, RT_PLATFORM_FAILURE, RT_UNSUPPORTED_PLATFORM

5.3 Shutdown

void rtExit()

This function MUST:

  • clean up all backend state
  • clean up all backend-allocated resources before it returns
  • invalidate all handles previously returned by the backend
  • transition the backend to the Uninitialized state
  • leave a subsequent call to rtInit() legal

Resource destruction MAY be deferred during normal execution, but no resource may remain uncleaned when rtExit() returns.

6. Free functions

Free functions are Rutile functions not associated with any resource or handle. Unless a function below explicitly states otherwise, free functions require the backend to be in the Initialized state; calling them before rtInit() is undefined behavior.

rt_error rtError()

Returns the current value of the thread-local error state.

Requirements: None

Pre-conditions: None

Post-conditions:

  • The thread-local error state is not modified.
  • The thread-local error message buffer is not modified.

Errors: None

Default: None

Notes: This function is defined behavior in both the Initialized and Uninitialized states. When called before any operation has executed on the calling thread, the returned value is uninitialized and its interpretation is undefined.

const char* rtErrorMessage()

Returns a pointer to the thread-local error message buffer.

Requirements: None

Pre-conditions: None

Post-conditions:

  • The returned pointer refers to thread-local storage and remains valid for the lifetime of the calling thread.
  • The buffer contents are only guaranteed to remain unchanged until the next operation executes on the same thread.
  • Before any message is written, and after rtClearError(), the buffer contains an empty string.

Errors: None

Default: None

void rtClearError()

Clears the thread-local error state and error message buffer.

Requirements: None

Pre-conditions: None

Post-conditions:

  • The thread-local error state is RT_SUCCESS.
  • The thread-local error message buffer contains an empty string.

Errors: None

Default: None

const char* rtGetName()

Returns a pointer to an implementation-defined string identifying the backend.

Requirements: None

Pre-conditions: None

Post-conditions:

  • The returned pointer MUST refer to static storage and MUST remain valid for the lifetime of the program.
  • The returned string MUST be encoded as UTF-8 and MUST be null-terminated.
  • The content of the string is implementation-defined.
  • The returned pointer MUST NOT change between calls.

Errors: None

Default: None

Notes: This function is defined behavior in both the Initialized and Uninitialized states.

7. Objects and resources

A resource is any backend-managed object. All resources are referenced through type-safe handles, and all resource behavior is expressed in terms of creation, state change, use, and destruction.

7.1 Resource types

buffer

a linear, untyped memory resource; interpretation as vertex data, index data, uniform data, or storage is determined by how it is bound

texture

a texel-addressable image resource used for sampling, storage, or attachment purposes

sampler

an immutable state object controlling how a texture is sampled (filtering, addressing, LOD)

framebuffer

a collection of texture attachments that rendering operations write into; may be user-owned or swapchain-owned

graphics_program

a linked GPU executable composed from a vertex stage and a fragment stage, each provided as RSIR bytecode

command_buffer

a recorded sequence of GPU commands prepared for later submission to a queue

swapchain

a presentation resource that manages presentable images for platform-specific display paths

queue

a submission endpoint that executes GPU work; queried from the backend via rtQueueQuery, not user-created or destroyed

7.2 Lifecycle

Every resource follows the lifecycle: Allocate, Configure, Use, Destroy. Allocation creates a resource and returns its handle. Configuration modifies resource state when the resource type supports it. Use operates on a valid resource according to its function contract. Destroy makes the handle invalid immediately.

Queue handles are an exception to this lifecycle. They are allocated and managed entirely by the backend at initialization time. The caller queries them via rtQueueQuery but does not create or destroy them. Queue handles are valid for the lifetime of the initialized backend and MUST NOT be passed to any destroy function.

7.3 Common rules

  • Destroying null MUST be a no-op.
  • Destroying an invalid non-null handle is undefined behavior.
  • A failed creation MUST NOT leak resources.
  • Resource state MUST NOT be unnecessarily duplicated. Where a platform already owns the authoritative value for a piece of state, the backend MAY use that platform state as the source of truth.

7.4 Generic create and destroy pattern

Each resource type defines its own typed handle TYPE_h and follows a resource-specific create and destroy function pattern. The common forms are shown below.

7.4.1 Create

TYPE_h rtTYPECreate()

On success, returns a valid handle and sets the error state to RT_SUCCESS. On failure, returns null and throws the appropriate error.

7.4.2 Destroy

void rtTYPEDestroy( TYPE_h handle )

Destroy MUST NOT fail. Destroying null MUST be a no-op. After destroy returns, the handle is immediately invalid. The backend MAY defer the physical cleanup of resources, but all resources MUST be cleaned up before rtExit() returns.

8. Validation layers

A validation layer is an optional intermediary that sits between the frontend and the backend, intercepting calls made through the operation model. A validation layer may export the same Rutile function names as a backend and forward calls to another loaded implementation.

Without a validation layer, calls flow from the frontend through loaded Rutile functions directly to the backend. With a validation layer inserted, calls flow from the frontend through loaded Rutile functions into the layer, which then forwards them through a second set of loaded Rutile functions to the backend.

Validation layers may:

  • validate calls against the rules defined in this specification before the backend sees them, and throw RT_IMPROPER_USAGE for violations
  • record Rutile function traffic for debugging, tracing, or tooling purposes
  • stop execution or emit diagnostics when frontend behavior violates required rules
  • observe, annotate, or adapt calls before forwarding them to a conforming implementation
  • modify opaque handles
  • introduce additional synchronization to make Rutile functions safe to call concurrently without external synchronization

Validation layers are fully composable. Backend implementations MAY rely on validation layers for additional diagnostics and are therefore not required to duplicate validation that is not mandated by this specification or the error model.

9. Resource-specific operations

The sections that follow define behavior specific to each resource type. The global lifecycle, handle, initialization, and error rules defined in sections 2 through 7 apply throughout and are not repeated unless a resource explicitly differs.

9.1 Swapchain

A swapchain is a presentation resource that manages one or more presentable images. Base Rutile does not define a swapchain creation function. Swapchains are created by extension-defined entry points such as rtWin32SwapchainCreate(). Platform bindings and presentation target associations are extension-defined rather than part of the core API.

  • The swapchain has no outstanding acquired framebuffer.

Errors: RT_PLATFORM_FAILURE, RT_DEVICE_LOST

Default: None

9.2 Texture

A texture is a texel-addressable resource. A texture handle identifies the texture object; texture storage and contents are part of its state.

Texture kind is described by rt_texture_type. This section defines RT_TEXTURE_1D, RT_TEXTURE_2D, RT_TEXTURE_3D, RT_TEXTURE_1D_ARRAY, and RT_TEXTURE_2D_ARRAY. Proxy textures, rectangle textures, cube textures, cube arrays, and other specialized kinds are not defined by this specification.

9.2.2 Storage definition

A newly created texture has no defined storage and kind RT_TEXTURE_UNKNOWN. Storage is defined when levels > 0 and undefined when levels == 0. When storage is undefined, the extent fields and internal format retain their current state values, but those values do not describe usable texture storage.

9.2.3 Extent model

Once storage is defined, every texture is described using four extents: width, height, depth, and layers. The values of unused extents are fixed as follows:

  • RT_TEXTURE_1D: height == 1, depth == 1, layers == 1
  • RT_TEXTURE_2D: depth == 1, layers == 1
  • RT_TEXTURE_3D: layers == 1
  • RT_TEXTURE_1D_ARRAY: height == 1, depth == 1
  • RT_TEXTURE_2D_ARRAY: depth == 1

9.2.4 Mip levels

Mip level 0 is the base level. Each successive mip level halves width, height, and depth, rounding down and clamping to a minimum of 1 for each active spatial dimension. The layer count does not change across mip levels.

9.2.5 Texture kinds

Texture kind is described by rt_texture_type. The value RT_TEXTURE_UNKNOWN is reserved as the 0 value and indicates that no texture kind has been defined. The supported kinds are RT_TEXTURE_1D, RT_TEXTURE_2D, RT_TEXTURE_3D, RT_TEXTURE_1D_ARRAY, and RT_TEXTURE_2D_ARRAY.

rtTextureData accepts any defined texture kind as its target argument. The interpretation of the z extent parameter depends on the target kind: for RT_TEXTURE_3D, z specifies the volume depth; for RT_TEXTURE_1D_ARRAY and RT_TEXTURE_2D_ARRAY, z specifies the layer count; for RT_TEXTURE_1D and RT_TEXTURE_2D, z must be 1.

9.2.6 Formats

Texture storage format is described by rt_format. The value RT_FORMAT_UNKNOWN is reserved as the 0 value and indicates that no format has been defined. Format suffixes are interpreted as follows: UNORM denotes unsigned normalized fixed-point storage; SFLOAT denotes signed floating-point storage; SINT denotes signed integer storage; UINT denotes unsigned integer storage.

Depth, stencil, and depth-stencil formats are attachment-oriented. Not every backend is required to support every format natively. Format capability rules are not yet defined by this specification.

10. Queues

10.1 Queue model

A queue is a submission endpoint that accepts and executes GPU work. Queues are backend-owned resources. They are not created or destroyed by the caller; they are queried by capability. The backend allocates an implementation-defined number of queues at initialization time. Queue handles remain valid for the lifetime of the initialized backend.

Multiple calls to rtQueueQuery with the same capability MAY return the same handle or different handles. The backend MAY distribute work across queues in any manner it chooses.

Because queue handles are not created by the caller, the create and destroy pattern in section 7.4 does not apply. Queue handles MUST NOT be passed to any destroy function.

10.2 Queue capability

Queue capability is described by rt_queue_capability. Capability values form a hierarchy: a queue that satisfies a higher capability also satisfies all lower ones. The hierarchy is: RT_QUEUE_GRAPHICS ⊇ RT_QUEUE_COMPUTE ⊇ RT_QUEUE_TRANSFER.

When a capability is requested, the returned queue satisfies at least the requested capability and MAY satisfy more. For example, requesting RT_QUEUE_COMPUTE MAY return a queue of kind RT_QUEUE_GRAPHICS if no dedicated compute queue is available.

NameValueGuaranteed capabilities
RT_QUEUE_TRANSFER1memory transfer operations
RT_QUEUE_COMPUTE2compute dispatch, memory transfer operations
RT_QUEUE_GRAPHICS3graphics draw calls, compute dispatch, memory transfer operations

rt_queue rtQueueQuery( rt_queue_capability capability )

Returns a handle to a backend-owned queue that satisfies the requested capability. The returned handle is valid for the lifetime of the initialized backend.

Requirements: None

Pre-conditions:

11. Synchronization

11.1 Timepoint model

A timepoint is a lightweight value that identifies a specific point on a queue's execution timeline. Every work-submitting function returns a timepoint by value. A timepoint can be used to query whether the associated work has completed on the CPU, to block the CPU until completion, or to express ordering constraints in a command stream.

Internally, each queue maintains a timeline semaphore. The value field of an rt_timepoint names a specific counter value on that semaphore. A timepoint is considered reached when the queue's timeline semaphore has advanced to or past that value.

A timepoint with queue == null and value == 0 is the null timepoint. The null timepoint is always considered reached. Any other combination of a null queue with a non-zero value, or a non-null queue with a zero value, is undefined behavior.

A caller that discards a returned timepoint and accesses the modified resource without any synchronization is in undefined behavior. The backend is not required to insert implicit synchronization.

11.2 rt_timepoint

rt_timepoint is a plain aggregate type:

struct rt_timepoint { queue_h queue; u64 value; }

Two timepoints are equal if and only if both queue and value are equal. A timepoint is null if queue == null && value == 0.

12. Command buffers

A command buffer is a recorded sequence of GPU commands that can be submitted to a queue for execution. Commands are recorded into the buffer on the CPU and executed asynchronously on the GPU after submission.

12.1 Lifecycle

A command buffer is in one of three states:

Initial

The command buffer has been created or has completed a previous submission. No commands are recorded. rtCommandBufferBegin may be called.

Recording

Commands are being recorded. rtCmd* functions may be called. rtCommandBufferEnd transitions to Executable.

Executable

Recording is complete. The buffer is ready for submission. rtQueueSubmit transitions back to Initial.

After rtQueueSubmit returns, the command buffer is in the Initial state and may be recorded into again. However, the GPU may still be executing the submitted commands. The caller MUST ensure the returned timepoint has been reached before recording new commands into the buffer.

The backend handles resource hazards automatically. No explicit barrier commands are required between operations on the same resource within a command buffer or across sequential submissions on the same queue. Cross-queue dependencies are expressed using rtCmdWaitTimepoint.
